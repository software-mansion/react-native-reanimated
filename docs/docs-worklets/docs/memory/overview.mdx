# Shared memory

Worklets let you run JavaScript on more than one thread. The moment you do that, you have to answer a question that single-threaded JavaScript has always let you ignore: **how do two pieces of code, running at the same time, look at the same data?**

This page explains why that question has no cheap answer in JavaScript, and why Worklets exposes three distinct shared memory primitives — the [Serializable](/docs/memory/serializable), the [Synchronizable](/docs/memory/synchronizable), and the [Shareable](/docs/memory/shareable) — instead of one.

## JavaScript is single-threaded on purpose

JavaScript was designed around a single-threaded execution model. One call stack, one event loop, one piece of code running at a time. That choice is load-bearing: it means a function that reads and mutates a variable cannot be interrupted halfway through by another function doing the same thing. There are no data races, no torn reads, no half-written objects. You never need a mutex to increment a counter.

The language leans on this guarantee everywhere. Closures capture variables by reference and expect those references to stay coherent. Objects are mutable, passed by reference, and shared freely across the program. None of this would be safe if two threads could touch the same heap at the same time — but in single-threaded JavaScript, it is safe by construction.

## What changes when you add a second runtime

React Native already runs JavaScript on more than one runtime. The **RN Runtime** runs your React tree and business logic on the JS thread. The **UI Runtime**, spawned by Worklets, runs high-priority work — gesture handling, animation frames — on the UI thread. You can also spawn additional **Worker Runtimes** via [createWorkletRuntime](/docs/threading/createWorkletRuntime) to move heavy computation off the JS thread entirely.

Each runtime is a fully independent JavaScript engine instance. Each one has its own heap, its own globals, its own GC, its own event loop. They run in parallel, on different threads, and — critically — **they cannot see each other's memory.** A variable you declared on the RN Runtime does not exist on the UI Runtime. An object you mutate on one side is invisible to the other. This is not a limitation of the Worklets library; it is a direct consequence of JavaScript's single-threaded design being preserved per runtime, which is what keeps each runtime fast and race-free internally.

So you have parallelism, but every runtime is an island. Useful work usually requires those islands to talk.

## Why shared memory abstractions are unavoidable

Once you accept that the runtimes have disjoint heaps, any cross-runtime communication reduces to the same question: **where does the value actually live, and who is allowed to touch it?** There is no way around this — it is the shape of the problem. You can pick a different answer, but you cannot avoid answering.

Worklets cannot "just" hand a JavaScript object from one runtime to another. A JavaScript object is a collection of pointers into one specific engine's heap. Its prototype chain, its hidden class, its property storage — all of it is engine-internal state that another runtime has no way to interpret. To cross the boundary, the value has to be reconstructed on the other side, or kept in a neutral location that both sides can reach through a controlled API.

That neutral location is C++. Every cross-runtime value in Worklets is ultimately a handle into a C++ object, accessed through [JSI](https://github.com/facebook/hermes/blob/main/API/jsi/jsi/jsi.h). The C++ side is thread-aware in ways that JavaScript is not: it can use mutexes, it can pin work to a specific runtime, it can copy bytes between heaps. This is the machinery that makes shared memory possible at all.

The reason there is more than one shared memory type is that **"shared" means different things at different costs.** Crossing the JSI boundary is not free. Locking is not free. Copying large objects is not free. Different use cases tolerate different tradeoffs, so Worklets exposes three primitives, each optimized for a different access pattern. Picking the right one matters — using a Synchronizable where a Serializable would do, or vice versa, will either be slow or wrong.

## The three shared memory types

Worklets distinguishes three kinds of shared memory, ordered here from the simplest to the most elaborate.

### Serializable — copy across the boundary

A [Serializable](/docs/memory/serializable) is a one-shot copy. A value on runtime A is serialized, transferred through C++, and reconstructed as an independent value on runtime B. After that, the two values have nothing to do with each other: mutating one does not affect the other, because there is no "other" — there are just two unrelated copies that happen to have started out equal.

Under the strictest definition, a Serializable is not shared memory at all, just a transport format. It is the right choice when you need to send a value once and do not expect the sender and receiver to keep observing each other's changes. Function arguments to [scheduleOnUI](/docs/threading/scheduleOnUI), return values from worklets — these are Serializables.

### Synchronizable — one value, many readers, lives in C++

A [Synchronizable](/docs/memory/synchronizable) is the first type that is genuinely shared. The actual value lives in C++, not in any JavaScript runtime. Every runtime holds only a reference. Every read or write crosses the JSI boundary, acquires a lock, touches the real value in C++, and returns.

Because the value is outside of every runtime, all runtimes see it symmetrically — no runtime is privileged, and a write from any of them becomes visible to all the others. The price is that every access pays for the JSI hop and the lock. Synchronizables are the right choice when the value is small, the access pattern is spread evenly across runtimes, and you need real consistency.

### Shareable — one value lives on one runtime, others reach in

A [Shareable](/docs/memory/shareable) is asymmetric shared memory. The value lives as a plain JavaScript value on one designated runtime, called the **Host Runtime**. Access on the Host is as cheap as any other JavaScript property access — no C++ hop, no lock. Other runtimes, called **Guest Runtimes**, hold a reference that knows the value is elsewhere; reading or writing from a Guest means scheduling work on the Host, waiting for it to read or mutate the real value, and serializing the result back.

This is the right choice when the access pattern is heavily skewed toward one runtime. Reanimated's Shared Value is the canonical example: the UI Runtime reads and writes the value dozens or hundreds of times per frame, and the RN Runtime pokes it occasionally to start an animation. Keeping the value on the UI side as a plain JavaScript value makes the hot path free; the slow path for the RN Runtime is an acceptable tradeoff.

## Choosing between them

A rough decision guide:

- **You are passing a value across runtimes once, and the other side does not need to observe further changes.** Use a Serializable. This is also what happens implicitly to worklet arguments and return values.
- **You need a value that multiple runtimes read and write with similar frequency, and you need them to see each other's updates.** Use a Synchronizable. Prefer primitives for best performance, since the value is copied across the JSI boundary on every access.
- **One runtime dominates access to the value, and other runtimes only reach in occasionally.** Use a Shareable, with the dominant runtime as the Host.

The rest of this section documents each primitive in detail, along with the APIs — [createSerializable](/docs/memory/createSerializable), [createSynchronizable](/docs/memory/createSynchronizable), [createShareable](/docs/memory/createShareable) — for producing them.
